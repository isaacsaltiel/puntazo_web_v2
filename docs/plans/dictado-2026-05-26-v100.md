## 4. Fase 3 — Ranking + Social v100

> **Greenfield.** No existe código previo en este repo. Diseñar como "version 100", no v0.01. Basado en research de sistemas existentes (Playtomic, PadelMix, UTR, MyTennis, etc.). Todo en Firestore desde el día 1 (no localStorage).

### Sub-fase 3.A — Motor de ranking (Glicko-2 robusto)

- **Algoritmo**: Glicko-2 completo con iteración de volatilidad (no Elo simple).
- **Calibración**: mínimo 3 partidos antes de bucket oficial. Antes: "🎯 Calibrando…".
- **RD visible**: rating como rango ("4.2 ± 0.4"), no número solo.
- **Conservative rating**: el bucket display penaliza incertidumbre. Rating 1900 con RD bajo = 🔥, mismo con RD alto = 🦅.
- **Margen de victoria (MOV)**: ganar 6-0 sube más que ganar 6-4. Curva suave `1 + log(1 + diff) × 0.12`.
- **Decay temporal**: si no juegas 30+ días, RD crece (sistema "olvida").
- **Anti-farm**: ganar repetidamente al mismo rival decae 1.0 → 0.5 → 0.25 (ventana 24h).
- **Audit trail**: cada partido guarda `ratingsBefore` / `ratingsAfter`. Si cambia el algoritmo, reprocesa el histórico.
- **Idempotencia**: re-procesar el mismo partido NO duplica cambios.
- **Buckets emoji escala 1.0–7.0** estilo Playtomic (🌱 🐣 🐥 🦅 🔥).
- **Página `mi-nivel.html`** (nueva): bucket, rango, posición + percentil, mejor pareja, rival más frecuente, mejor puntazo, sparkline histórico, sticker compartible.
- **Cloud Function batch** para recompute periódico (en caso de cambios al algoritmo).

### Sub-fase 3.B — Sistema de identidad robusto

- Perfil con foto, displayName vs nombre real.
- **Claim cruzado** (caso clave): usuario nuevo se registra; su nombre estaba en partidos previos como string suelto. Puede hacer "ese soy yo" sobre slots históricos. El creador del partido autoriza el claim. Histórico de partidos retroactivos se le adjudica.
- **Prevención de impersonación**: link a Google obligatorio (ya está vía Firebase Auth). Verificación opcional por phone para usuarios con muchos claims.
- **Account merge**: si un usuario terminó con 2 cuentas (anónima + después logueada), flujo de merge.
- **Account recovery**: por email o por Google.
- **Borrar cuenta**: opción visible. Borra datos personales, deja partidos como "Usuario eliminado".

### Sub-fase 3.C — Grupos / Ligas

- Crear grupo manual con nombre, descripción, foto opcional.
- El creador es admin. Puede asignar co-admins.
- Link de invitación compartible (WhatsApp / copiar link).
- Unirse al grupo poniendo nombre (si no logueado) o vinculando uid (si logueado).
- Admins pueden: kickear miembros, cambiar nombre del grupo, asignar otros admins.
- **Auto-descubrimiento**: si juegas 3+ partidos con las mismas N personas, sistema sugiere "¿quieres armar grupo con ellos?".
- **Ranking interno del grupo**: filtrar partidos donde TODOS los jugadores son miembros del grupo. Bucket Playtomic separado del global.
- **Liga formal** (futuro): round-robin de N semanas/meses entre miembros, calendario, brackets.
- Tipos especiales de grupo: "Liga del residencial", "Liga del club" (auto-add por geolocalización opcional).

### Sub-fase 3.D — Amigos

- Enviar / aceptar / rechazar solicitud de amistad.
- Ver perfil de amigo: sus partidos públicos, ranking, stats, head-to-head contigo.
- Buscar usuarios por nombre / email.
- **Head-to-head histórico** contra cualquier rival (no solo amigo): "Has jugado 5 vs Pedro, ganaste 3".
- Bloquear / reportar usuario.
- Lista de amigos visible en perfil.

### Sub-fase 3.E — Autocomplete de jugadores recientes

- Al escribir nombres en `mi-partido.html` (form previo, edición durante partido, terminar) o en `registrar-partido.html`: dropdown con sugerencias.
- Fuente: pool de `displayName`s de los últimos N partidos del usuario.
- Si el usuario ya está logueado al jugador (uid), pre-llenar con su displayName.
- Si el nombre es uno frecuente con uid asociado, mostrar avatar chico al lado.

### Sub-fase 3.F — Invitaciones multi-target

- **Invitar a Puntazo**: link genérico con landing page de install/sign-in. Si el invitador es identificable, mostrar "[Nombre] te invitó a Puntazo".
- **Invitar a partido**: link específico al matchId. Si el invitado abre el link y no se ha unido al partido, puede claimar slot. OG tags para preview lindo en WhatsApp.
- **Invitar a ver clip**: link a clip específico (ya existe vía explorar.html, mejorar con OG tags).
- **Invitar a grupo**: link al grupo, abre flow de aceptación.
- Cada link unfurleable en WhatsApp con foto + título + descripción.

### Sub-fase 3.G — Notificaciones

In-app primero, push después.

Eventos:
- Te claimaron un slot en un partido tuyo (creador autoriza/rechaza).
- Un amigo se unió a un partido tuyo.
- Alguien te agregó a un grupo.
- Tu ranking cambió de bucket.
- Alguien te retó a partido (futuro).
- Te invitaron a Puntazo (cuando el invitador es identificable).
- Tu clip está listo (puntazo pendiente → visible).

### Sub-fase 3.H — Privacy + manejo de cuentas

- Perfil: público / amigos / privado.
- Quién ve tus clips: todos / amigos / nadie (por default todos).
- Account recovery por Google.
- Account merge si terminó con 2.
- Borrar cuenta (soft-delete, datos personales removidos, partidos quedan como "Usuario eliminado").

### Sub-fase 3.I — Disputas y edición post-cierre

- Ventana de edición del marcador después de cerrar partido (24h por default, configurable).
- Sistema de disputa: si 2 jugadores claiman versiones diferentes del marcador, el creador del partido decide. Si nadie decide en X tiempo, marcador queda como está al cierre.
- Log audit de toda edición.

### Sub-fase 3.J — Research de sistemas existentes

Investigar y sintetizar:
- **Playtomic**: ranking por nivel 1.0-7.0, matchmaking, sistema social.
- **PadelMix**: ligas y torneos.
- **UTR (Universal Tennis Rating)**: algoritmo robusto, score-based.
- **MyTennis / iTennis**: identidad y grupos.
- **Strava**: gamificación, social feed, sharing.

Identificar:
- Mejores prácticas a copiar.
- Anti-patterns a evitar.
- Qué hace Puntazo diferente (clips automáticos + rotaciones sociales + ranking serio).

### Sub-fase 3.K — Reglas Firestore

Para todas las colecciones nuevas:
- `users/{uid}` — perfil
- `users/{uid}/friends/{friendUid}` — amistades (con status)
- `friend_requests/{requestId}` — solicitudes pendientes
- `groups/{groupId}` — grupos/ligas
- `groups/{groupId}/members/{uid}` — membresías
- `group_invites/{inviteId}` — invitaciones pendientes
- `match_invites/{inviteId}` — invitaciones a partido específico
- `notifications/{uid}/items/{notifId}` — notificaciones in-app
- `disputes/{matchId}` — disputas activas
- `claims_history/{matchId}/{slotIdx}` — historial de claims (para claim cruzado)

Rules con tests.

## 5. Pendientes y consideraciones extra (Isaac no mencionó pero importantes)

- **Performance**: collectionGroup queries pueden ser caras. Para "Mis partidos" usar el approach que ya tenemos (collectionGroup `claims` where uid==user.uid).
- **Offline mode**: muchos usuarios juegan en clubes con WiFi malo. Cache local con Firestore offline persistence.
- **i18n**: por ahora solo español. Pero estructurar strings centralizados.
- **Accesibilidad**: contraste, alt texts, navegación con teclado.
- **Bot/cheat detection**: cuentas que farmean a sí mismas para subir ranking. Detectar y flaggear.
- **Onboarding**: primer-uso debe ser claro. Tutorial chico opcional.
- **Search Engine**: que el perfil público sea indexable (OG tags, schema.org).
- **Avatares**: subida a Firebase Storage, redimensión client-side a 256×256.
- **Stats avanzadas**: mejor / peor pareja, hora del día con mejor rendimiento, días de la semana.
- **Achievements / badges**: 10 partidos jugados, primera victoria, racha 5, etc.
